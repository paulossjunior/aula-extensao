# Aula 01 — Desenvolvimento de Produto Baseado em Hipótese

**Data:** 09/03/2026 | **Horário:** 11h00 | **Local:** Sala 207

---

Nesta aula inauguramos o curso apresentando o **[processo de desenvolvimento de produto baseado em hipótese](https://theleanstartup.com/principles)**. Esta é uma abordagem ágil usada para validar ideias por meio de testes empíricos, reduzindo riscos e focando em evidências. Em vez de construir diretamente uma solução, partimos de um problema observado e testamos suposições antes de investir em desenvolvimento — seguindo o ciclo *Build → Measure → Learn* do Lean Startup. Ao longo do curso, cada equipe conduzirá esse processo para criar seu próprio produto.

---

## Discovery do Projeto

Nesta primeira etapa do processo, o foco é **entender o problema antes de propor qualquer solução**. Os três pilares metodológicos que guiam esse processo são:

- **Human-Centered Design (HCD) — Design Centrado no Humano**
  Abordagem criativa que coloca as necessidades e comportamentos reais das pessoas no centro do processo. Antes de construir qualquer coisa, é preciso *observar, empatizar e aprender* com quem tem o problema.
  [Leia a introdução ao HCD — IDEO](https://www.ideou.com/pages/design-thinking)

- **Product Requirements Document (PRD) — Documento de Requisitos de Produto**
  O PRD consolida o problema, os objetivos, o público-alvo, os requisitos e as métricas de sucesso. É o ponto de partida para alinhar produto, design e engenharia.
  [Guia completo de PRD — Product School](https://productschool.com/blog/product-management-2/product-requirements-document/)

- **Design Sprint — Processo de Validação Acelerada**
  Metodologia criada pelo Google Ventures para responder perguntas críticas de negócio em 5 dias: mapeamento, esboço, decisão, prototipação e teste com usuários reais.
  [Design Sprint oficial — GV (Google Ventures)](https://www.gv.com/sprint/)

- **Desenvolvimento Orientado a Hipóteses (Conceitos e Prática)**
  Referências para aprofundar na gestão de produtos orientada por evidências e experimentação:
  - [Gestão de Produtos Orientado a Hipóteses — UX Collective Brasil](https://brasil.uxdesign.cc/gest%C3%A3o-de-produtos-orientado-a-hip%C3%B3teses-acfc9a7e3162)
  - [Desenvolvimento Orientado por Hipóteses: Guia de Gestão — Awari](https://awari.com.br/desenvolvimento-orientado-por-hipoteses-um-guia-para-uma-gestao-de-produtos-mais-inteligente/)
  - [Desenvolvimento Orientado por Hipóteses — Somos Tera (Medium)](https://medium.com/somos-tera/desenvolvimento-orientado-por-hipotesis-b175b34a6991)
  - [Vídeo: O que é desenvolvimento por hipóteses? — YouTube](https://www.youtube.com/watch?v=y1atheNugVQ)
  - [Hipótese de um Projeto: Etapas Práticas — Compre TCC](https://compretcc.com/hipotese-de-um-projeto-2/)

!!! tip "Dica de Ouro: Como formular uma hipótese"
    Uma hipótese clara deve seguir a estrutura: **"Se [ação], então [resultado esperado] para [usuário]"**. Isso evita criar soluções sem conexão com necessidades reais.

!!! note "Leitura recomendada antes da aula"
    Leia o resumo do [The Mom Test](http://momtestbook.com/) — como conversar com potenciais clientes sem receber respostas enviesadas.

## Cuidado: Erros Comuns na Identificação de Problemas

Identificar problemas e oportunidades é crucial, mas alguns erros frequentes podem comprometer a validação de ideias:

- **Falta de Dados Concretos**
  Muitos problemas são identificados com base apenas em intuição. É essencial buscar evidências reais e métricas dos usuários.
  [Gestão de Produtos Orientada a Hipóteses — brasil.uxdesign](https://brasil.uxdesign.cc/gest%C3%A3o-de-produtos-orientado-a-hip%C3%B3teses-acfc9a7e3162)

- **Visão Superficial (Focar em Sintomas)**
  Tratar apenas sintomas sem analisar as causas raiz gera soluções paliativas. Use ferramentas como os *5 Porquês*.
  [Oportunidades de Melhoria — siteware.com](https://www.siteware.com.br/blog/qualidade/oportunidades-de-melhoria/)

- **Equipe Inadequada (Falta de Stakeholders)**
  Não envolver usuários finais ou áreas operacionais gera visões enviesadas e soluções desconectadas da realidade.
  [Erros Comuns em Resolução de Problemas — LinkedIn](https://pt.linkedin.com/pulse/os-erros-mais-comuns-em-resolu%C3%A7%C3%A3o-de-problemas-thayc-marinho)

- **Confundir Correlação com Causalidade**
  Assumir que eventos correlacionados são causais distorce as oportunidades. Testes rigorosos são necessários.
  [Erros Comuns em Resolução de Problemas — LinkedIn](https://pt.linkedin.com/pulse/os-erros-mais-comuns-em-resolu%C3%A7%C3%A3o-de-problemas-thayc-marinho)

- **Ignorar o Contexto Externo**
  Falhar em analisar o mercado e a concorrência leva a oportunidades irreais.
  [Erros no Planejamento Estratégico — sebrae.com](https://sebrae.com.br/sites/PortalSebrae/ufs/pe/artigos/8-erros-mais-comuns-no-planejamento-estrategico-e-como-evita-los,63511c3d3dfc3910VgnVCM1000001b00320aRCRD)

## Tarefas

### Tarefa 1 — Identificar o Problema

**Duração estimada:** 20 min
**Formato:** Individual → compartilhamento em grupo

Cada participante deve descrever **um problema que observa no seu dia a dia** e que acredita que poderia ser resolvido com um produto ou serviço.

Use as perguntas guia:

- Quem tem esse problema?
- Com que frequência ele acontece?
- Quais são as consequências de não resolver?
- Como as pessoas lidam com isso hoje?

**Entregável:** Um parágrafo descrevendo o problema observado.

---

### Tarefa 2 — Mapear Soluções Existentes

**Duração estimada:** 25 min
**Formato:** Duplas ou trios

Com base no problema identificado, pesquise se já existem **produtos ou serviços** que tentam resolvê-lo (total ou parcialmente).

1. Liste ao menos **3 soluções existentes** (pode ser app, serviço, produto físico, processo manual)
2. **Categorize o produto:** SaaS, marketplace, produto físico, serviço profissional, etc.
3. Identifique **2 concorrentes diretos** e **1 concorrente indireto**
4. Anote pelo menos **3 indicadores de sucesso do produto** que esse tipo de solução costuma monitorar (ex: taxa de retenção, NPS, churn, GMV, DAU/MAU, LTV)
5. Pesquise se há benchmarks ou médias de mercado para esses indicadores

| Item                    | Descrição                        |
|-------------------------|----------------------------------|
| Categoria               |                                  |
| Concorrente direto 1    |                                  |
| Concorrente direto 2    |                                  |
| Concorrente indireto    |                                  |
| Indicador de sucesso 1  |                                  |
| Indicador de sucesso 2  |                                  |
| Indicador de sucesso 3  |                                  |
| Benchmark de mercado    |                                  |

**Entregável:** Tabela preenchida + breve justificativa das escolhas.

---

### Tarefa 3 — Apresentação Rápida (*pitching*)

**Duração estimada:** 20 min
**Formato:** Grupo completo

Cada dupla apresenta em **até 3 minutos**:

1. Qual o problema?
2. Quem já tenta resolver?
3. Qual o gap que identificaram?

A turma dá feedback usando o modelo **"Brilha / Melhora"**: aponte um ponto forte e uma sugestão de melhoria.

**Entregável:** Apresentação oral (sem slides necessários).
